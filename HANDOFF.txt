1. WHAT THIS IS
-------------------------------------------------------------------
- A native macOS menubar app that talks to Samsung Galaxy Buds (the
  author's pair: Galaxy Buds 3 FE) directly over Bluetooth, with no
  Samsung phone or Galaxy Wearable app involved.
- Goal: an AirPods-style experience on the Mac - battery for Left/Right/
  Case, ANC mode, and EQ preset from the menubar, plus an AirPods-style
  "Connect" HUD that pops when you open the case / reconnect.
- Platform: macOS 14+ (Sonoma). Menubar-only app (LSUIElement, no Dock icon).
- Audience: personal use by the author (Masimba), released as open source
  under GPL-3.0-or-later. Protocol reverse-engineered from live captures;
  framing follows the well-known ThePBone/GalaxyBudsClient project.


2. TECH STACK & ARCHITECTURE
-------------------------------------------------------------------
LANGUAGE / FRAMEWORK
- Swift (swift-tools-version 5.10), built with Swift Package Manager.
  NOT Electron, NOT Tauri - 100% native Swift.
- UI: SwiftUI components hosted in AppKit (NSStatusItem menubar + NSPopover
  panel + NSPanel HUD). 3D earbud view uses SceneKit/RealityKit-style USDZ
  loading with a procedural fallback.
- Bluetooth: TWO Apple frameworks used together:
    * CoreBluetooth (BLE) -> passive scan for Samsung manufacturer adverts
      (company ID 0x0075); used only to detect "lid opened" via payload-
      length jump. Contents are encrypted; we never decode them.
    * IOBluetooth (Classic BT) -> the real control path: connect/disconnect
      notifications, SDP service query, and the RFCOMM channel we read
      battery from and write commands to.
- No third-party dependencies. No Package.resolved (nothing to resolve).

ENTRY POINT & FLOW
- BuddyApp (@main) -> AppDelegate.applicationDidFinishLaunching:
    1. prints build banner + runs encoder selfTest
    2. parses CLI flags (--channel N, --send AABB)
    3. sets activation policy .accessory (menubar-only)
    4. builds BudsViewModel.mock(), creates MenuBarController
    5. starts BTConnectionMonitor (classic BT) + AdvertScanner (BLE)
- On classic-BT connect: shows HUD, performs SDP query, finds GEARMANAGER
  service (match by UUID 2e73a4ad..., channel 27), opens RFCOMM channel.
- Incoming RFCOMM bytes -> BudsStreamParser.feed() -> BudsMessage(s) ->
  BudsStatusDecoder -> BudsViewModel.apply() updates the SwiftUI battery rings.
- ANC/EQ clicks in the panel -> BudsViewModel.onControl -> AppDelegate.sendControl
  -> BudsCommand.noiseControl/equalizer -> BTConnectionMonitor.send().

CONVENTIONS
- Diagnostic logging is the primary debugging tool: run from Terminal to see
  [BLE], [BT], [SDP], [RFCOMM], [SEND], [MSG], [MSG?], [BATTERY], [SELFTEST].
- Build identity tag lives in AppDelegate.buildTag (currently "reconnect-12");
  bump it when iterating so the banner tells you which build is running.
- Opcodes/wire mappings that are unconfirmed are marked "CANDIDATE" and made
  `var` so they're trivial to retune once verified on hardware.
- IMPORTANT - there are TWO protocol stacks in the repo. The LIVE one is
  BudsProtocol/Encoder+StreamParser+BudsStatus (SOM 0xFD/EOM 0xDD), driven
  directly by AppDelegate. The OLD scaffold (Frame.swift/BudsFrameCodec with
  SOM 0xFE, RFCOMMChannel actor, BudsManager) is leftover dead code from the
  initial commit and is NOT referenced by the running app. Don't be misled
  by it; consider deleting it (see KNOWN ISSUES).

5. WHERE TO PICK UP   *** MOST IMPORTANT ***
-------------------------------------------------------------------
The read path is done. The next milestone is making ANC + EQ actually CONTROL
the buds (true two-way control), which requires a short on-hardware session.
You need the physical Galaxy Buds 3 FE paired to the Mac to do this.

STEP 1 - Confirm nothing regressed (5 min, with buds connected):
  Build + run from Terminal (see section 7) and confirm the six checks in
  TESTING.md section 1: build banner, [SELFTEST] encoder = PASS, channel 27
  opens (status=0), [BATTERY] matches your phone, panel rings correct, HUD pops.

STEP 2 - Find the ANC-mode status byte (passive, safe) - TESTING.md section 2:
  With buds in your ears and Buddy running, change the noise mode ON THE PHONE
  (Off -> Ambient -> ANC -> Adaptive), copying the "[MSG] id=0x61 payload=" line
  after each. The byte that changes is the ANC offset. Then ADD that offset to
  BudsStatusDecoder.decode() in Sources/BudsProtocol/BudsStatus.swift (extend
  BudsStatus with an `ancMode` field and decode it), and have
  BudsViewModel.apply() in Sources/BudsUI/BudsViewModel.swift set
  device.ancMode so the segmented control reflects reality.

STEP 3 - Find the ANC *send* opcode (active probe) - TESTING.md section 3:
  Click ANC in the panel first; if nothing changes, sweep candidates from the
  CLI (worst case = harmless no-op, CRC is valid):
      ./.build/debug/Buddy.app/Contents/MacOS/Buddy --send 7901
      ...try 7900/7902 and other msgIDs 7801/9801/9a01 with payloads 00/01/02
  Watch your ears + the [MSG] 0x61 ANC byte. The (msgID,payload) that flips the
  mode is the real command. Then update Sources/BudsProtocol/Commands.swift:
  set BudsOpcode.noiseControl and fix the value mapping in
  AppDelegate.ancWireValue() (Sources/Buddy/AppDelegate.swift, ~line 82).

STEP 4 - Same for EQ - TESTING.md section 4: confirm/fix BudsOpcode.equalizer

7. HOW TO RUN & DEVELOP
-------------------------------------------------------------------
PREREQS: macOS 14+. Swift toolchain (Xcode Command Line Tools) is enough to
BUILD and RUN. FULL Xcode is needed only to run the test target locally and for
a notarized release. A real Bluetooth Mac + paired Galaxy Buds for live testing.

WHY A BUNDLE: CoreBluetooth needs a real, signed .app with an Info.plist
Bluetooth usage string to get TCC permission - a bare SPM binary won't get the
prompt. So always run via the bundle, not `swift run`.

BUILD + RUN (from the project root):
    ./Packaging/make-app.sh debug
    ./.build/debug/Buddy.app/Contents/MacOS/Buddy
  (make-app.sh does: swift build -c debug, assembles Buddy.app, copies the
   Info.plist + SwiftPM resource bundles, then ad-hoc codesigns it.)
  Release variant: ./Packaging/make-app.sh release

  Run from Terminal (not by double-clicking) so you see the diagnostic log.
  An "earbuds" icon appears in the menubar; click it for the panel. Ctrl-C quits.
  First run: grant Bluetooth permission in System Settings > Privacy & Security.

USEFUL CLI FLAGS:
    --channel N     force a specific RFCOMM channel (e.g. --channel 27)
    --send AABB     send raw frame (msgID AA, payload BB) ~0.6s after connect,
                    repeatable - the opcode-probing harness used in section 5

TESTS:
    swift test      (needs full Xcode; runs on CI via .github/workflows/ci.yml)

ENV VARS / SECRETS: none. No API keys, no .env, no external services. Nothing
to configure beyond granting the Bluetooth permission.

SIGNING/NOTARIZATION: currently ad-hoc signed only (codesign --sign -). For a
real distributable release you'd need a Developer ID cert + notarization; not
set up yet.
